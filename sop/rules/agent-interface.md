# 에이전트 간 인터페이스 (Agent Interface)

## 목적

에이전트 간 결과 전달 형식, 실행 실패 시 대응 절차, 역할별 검토 체크리스트를 정의한다.

---

## 1. 에이전트 산출물 표준 형식

모든 Researcher 에이전트는 아래 템플릿에 맞춰 결과를 산출한다. Manager/QA가 파싱할 수 있도록 구조를 통일한다.

```markdown
---
agent_role: researcher
section: "{조사 항목 번호}.{하위 항목}"
gene: "{대상 유전자}"
disease: "{대상 질병}"
species: "{조사 종}"
timestamp: "YYYY-MM-DD HH:MM"
---

## 조사 결과

### 핵심 발견 (Key Findings)
- (발견 1) [Evidence Level] — (출처 URL)
- (발견 2) [Evidence Level] — (출처 URL)

### 상세 내용 (Details)
(서술형 내용)

### 검색 로그 (Search Log)
| # | 검색 쿼리 | 대상 DB | 결과 건수 | 유효 결과 |
|---|----------|---------|----------|----------|
| 1 | ... | ... | ... | Y/N |

### Manager 전달 사항
- (해석이 필요한 부분, 판단 요청 사항 등)
```

---

## 2. Manager 통합 산출물 형식

Manager 에이전트는 하위 Researcher 결과를 통합할 때 아래 형식을 사용한다.

```markdown
---
agent_role: manager
section: "{조사 항목 번호}"
sub_sections: ["{하위 항목 1}", "{하위 항목 2}", ...]
---

## 통합 결과

### 항목 요약
(하위 항목 결과를 종합한 1~3문장 요약)

### 하위 항목별 결과 요약
| 하위 항목 | 근거 수준 | 핵심 발견 |
|-----------|----------|----------|
| ... | [Level] | ... |

### 통합 해석
(하위 항목 간 관계, 일관성, 모순점 서술)

### QA 전달 사항
- (교차검증이 필요한 출처, 불확실한 판단 등)
```

## 3. 실행 실패 시 Fallback 절차

에이전트가 할당된 작업을 완료하지 못하는 경우, 아래 절차에 따라 대응한다.

### 실패 유형별 대응

| 실패 유형 | 원인 예시 | Fallback 절차 |
|-----------|----------|--------------|
| **도구 권한 오류** | WebSearch/WebFetch 권한 거부 | 1. 즉시 상위 에이전트에 보고 2. 상위 에이전트가 동일 검색을 직접 수행 |
| **데이터베이스 접근 불가** | 서버 다운, 403/404 오류 | 1. 대체 DB로 전환 (예: NCBI → Ensembl) 2. 3회 재시도 후 실패 시 보고 |
| **검색 결과 과다** | 1,000건 이상 비특이적 결과 | 1. 검색 쿼리를 더 구체적으로 조정 2. 별칭 + 질병 + 조직 조합으로 좁힘 |
| **검색 결과 전무** | 0건 | 1. 별칭 변경 후 재검색 2. 최소 검색 요건 충족 확인 후 "근거 불충분" 판정 |
| **타임아웃** | 처리 시간 초과 | 1. 검색 범위를 축소하여 재시도 2. 부분 결과라도 산출 |

### 보고 형식

실패 시 아래를 포함한 실패 보고서를 산출한다.

```markdown
---
agent_role: researcher
status: FAILED
failure_type: "{실패 유형}"
---

## 실패 보고
- **할당 항목**: {항목명}
- **실패 유형**: {도구 권한 오류 / DB 접근 불가 / ...}
- **시도 내역**: {수행한 시도 목록}
- **부분 결과**: {있는 경우 기재}
- **권장 대응**: {대체 DB 추천, 쿼리 제안 등}
```

## 4. 역할별 검토 체크리스트

### Manager 검토 체크리스트

Manager는 Researcher 결과를 수령한 후 아래 항목을 확인한다.

- [ ] 모든 핵심 발견에 출처 URL이 첨부되어 있는가?
- [ ] 증거 등급이 [evidence-grading.md](evidence-grading.md) 기준에 맞게 부여되었는가?
- [ ] 종(species) 구분이 명확한가?
- [ ] 2차 요약 자료(GeneCards 등)가 사용되지 않았는가?
- [ ] 추측성 서술이 포함되어 있지 않은가?
- [ ] 동물실험 결과가 인간 사실로 서술되지 않았는가?
- [ ] 최소 검색 요건([insufficient-evidence.md](insufficient-evidence.md))이 충족되었는가?
- [ ] Narrow-down 조직 우선 서술이 적용되었는가?
- [ ] 검색 로그가 포함되어 있는가?

### QA 검토 체크리스트

QA 에이전트는 Manager 통합 결과에 대해 아래를 확인한다.

- [ ] 각 핵심 주장을 출처 URL에서 직접 확인 가능한가? (샘플링 검증: 전체의 30% 이상)
- [ ] 출처에 기재된 내용과 보고서 서술이 일치하는가?
- [ ] 출처의 종(species)이 보고서의 종 구분과 일치하는가?
- [ ] 증거 등급이 실제 근거 수준과 부합하는가?
- [ ] "근거 불충분" 판정 항목의 검색 범위가 적절한가?
- [ ] 인접 유전자좌 혼동이 없는가?

### QA 불합격 시 절차

| 불합격 사유 | 조치 |
|------------|------|
| 출처-내용 불일치 | 해당 항목 Researcher에게 재조사 요청 |
| 증거 등급 부적절 | Manager에게 재평가 요청 |
| 검색 범위 부족 | 해당 항목 추가 검색 후 재제출 |
